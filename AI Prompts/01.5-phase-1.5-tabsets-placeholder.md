#### `vbotTabPlaceholder.js`

```javascript
import { LightningElement, api } from 'lwc';

export default class VbotTabPlaceholder extends LightningElement {
    @api tabLabel = 'Coming soon';
    @api variant = 'auto';
    @api phaseLabel = 'Phase 2';

    get resolvedVariant() {
        if (this.variant && this.variant !== 'auto') {
            return this.variant;
        }
        const label = (this.tabLabel || '').toLowerCase();
        if (label.includes('service') || label.includes('warranty')) return 'service-warranty';
        if (label.includes('upgrade') || label.includes('sales')) return 'upgrades';
        if (label.includes('connected') || label.includes('asset') || label.includes('telemetry')) return 'connected-asset';
        return 'generic';
    }

    get isServiceWarranty() { return this.resolvedVariant === 'service-warranty'; }
    get isUpgrades() { return this.resolvedVariant === 'upgrades'; }
    get isConnectedAsset() { return this.resolvedVariant === 'connected-asset'; }
    get isGeneric() { return this.resolvedVariant === 'generic'; }
}
```

## Track B – FlexiPage Tabsets

Approach in **zwei Schritten**: zuerst Metadata-API versuchen mit explizitem Tabset-XML, bei Misserfolg auf UI-Retrieve umsteigen.

### B.1 – Metadata-Approach (zuerst probieren)

Die FlexiPage-XML-Struktur für ein Tabset verwendet `flexipage:tabset` als `componentInstance` in der Main-Region. Wichtig: die `tabs`-Property ist ein JSON-String, der in XML escapt werden muss. Die `tabsContent`-Property referenziert separate FlexiPage-Regions, deren Namen identisch zur Tab-ID sein müssen.

#### Validate-Deploy

```bash
sf project deploy validate \
  --metadata "FlexiPage:VBOT_WorkOrder_Record_Page,FlexiPage:VBOT_Asset_Record_Page" \
  --target-org <alias>
```

**Wenn Validate sauber durchläuft:** Deploy starten und Track B als erledigt markieren.

**Wenn Validate Fehler wirft – typische Probleme:**

| Fehler | Ursache | Fix |
|---|---|---|
| `Cannot resolve component flexipage:recordDetailPanel` | falscher Component-Name | versuche `flexipage:recordFullSizeForm` oder `flexipage:recordHighlightsAndDetails` |
| `Cannot resolve component runtime_sales_activities:activitiesPanel` | nicht in dieser Org/Edition | weglassen, später per UI nachziehen |
| `Cannot resolve component runtime_chatter:feedContainer` | nicht deploybar | weglassen, später per UI nachziehen |
| `Invalid tabs JSON` | XML-Escape-Problem | `&amp;` für `&` prüfen; JSON-Quotes als reine `"`, nicht `&quot;` |
| `Region name 'detailTab' not referenced by tabset` | Region-Name muss in `tabs`-JSON als `id` auftauchen | abgleichen |

**Strategie bei Komponenten-Fehlern:** Entferne nur die fehlenden Komponenten, behalte das Tabset.

### B.2 – UI-Fallback (nur falls B.1 final scheitert)

Wenn die Metadata-Approach nach 2 Versuchen nicht zum Deploy führt:

1. Setup → Object Manager → Work Order → Lightning Record Pages → `VBOT Work Order Record Page` → **Edit**
2. Im App Builder:
   - **Change Layout** → wähle ein 3-Region-Template (Header + Main + Right Sidebar)
   - In **Main**: „Tabs"-Komponente reinziehen
   - Tab 1 umbenennen → "Details" → Inhalt: Record Detail + Related Lists (Asset, WO Line Items)
   - Tab 2 hinzufügen → Custom Label "Service & Warranty" → Inhalt: `c:vbotTabPlaceholder` mit `tabLabel = "Service & Warranty"`, `variant = "service-warranty"`, `phaseLabel = "Phase 2"`
   - Tab 3 hinzufügen → Custom Label "Upgrades" → Inhalt: `c:vbotTabPlaceholder` mit entsprechenden Properties
   - In **Sidebar**: Activities + Chatter
3. **Save → Activate → Org Default → Desktop and phone**
4. Analog für Asset Record Page (zwei Tabs: Details + Connected Asset)
5. Retrieve:

```bash
sf project retrieve start \
  --metadata "FlexiPage:VBOT_WorkOrder_Record_Page,FlexiPage:VBOT_Asset_Record_Page" \
  --target-org <alias>
```

## Track C – Drei fehlende ListViews

Approach: **Metadata-First mit verifizierten Tokens**, UI-Fallback nur bei finalem Misserfolg.

### Token-Referenz

Aus Discovery: `ACCOUNT.NAME`, `ASSET.NAME` funktionieren. Daraus abgeleitete Konvention für ListView-XML-Column-Tokens:

- **Standard-Felder eines Standardobjekts:** all-caps mit Underscore (`NAME`, `STATUS`, `WORK_ORDER_NUMBER`, `CONTRACT_NUMBER`, `START_DATE`, `END_DATE`, `PRODUCT_CODE`, `IS_ACTIVE`, `SUBJECT`)
- **Custom-Felder:** API-Name mit `__c` (`VBOT_Demo_Scenario__c`, `VBOT_Tier__c`, `VBOT_Tier_Label__c`, `VBOT_Max_Area_Sqm__c`)
- **Relationship-Felder:** `RELATION.NAME` (`ACCOUNT.NAME`, `ASSET.NAME`)

### C.1 – `VBOT_Products` (Product2)

```xml
<?xml version="1.0" encoding="UTF-8"?>
<ListView xmlns="http://soap.sforce.com/2006/04/metadata">
    <fullName>VBOT_Products</fullName>
    <columns>NAME</columns>
    <columns>PRODUCT_CODE</columns>
    <columns>VBOT_Tier_Label__c</columns>
    <columns>VBOT_Max_Area_Sqm__c</columns>
    <columns>IS_ACTIVE</columns>
    <filterScope>Everything</filterScope>
    <filters>
        <field>PRODUCT_CODE</field>
        <operation>startsWith</operation>
        <value>VBOT</value>
    </filters>
    <label>VBOT Products</label>
</ListView>
```

### C.2 – `VBOT_Service_Contracts` (ServiceContract)

Analog mit Filter `NAME startsWith VBOT`.

### C.3 – `VBOT_Work_Orders` (WorkOrder)

Analog mit Filter `VBOT_Demo_Scenario__c notEqual ""`.

### Validate-Deploy (einzeln, damit Fehler isoliert sichtbar werden)

```bash
sf project deploy validate --metadata "ListView:Product2.VBOT_Products" --target-org <alias>
sf project deploy validate --metadata "ListView:ServiceContract.VBOT_Service_Contracts" --target-org <alias>
sf project deploy validate --metadata "ListView:WorkOrder.VBOT_Work_Orders" --target-org <alias>
```

**Wenn ein Validate scheitert:** Token-Variante austauschen (z. B. `PRODUCT_CODE` → `Product2.ProductCode` → `ProductCode`).

### C.4 – UI-Fallback (nur wenn nach Token-Variation immer noch nicht deploybar)

1. App Launcher → Service oder Field Service Console → Product2-Tab
2. Drittes Icon oben rechts: ListView-Controls → „New"
3. Name: `VBOT Products`, API Name: `VBOT_Products`, Visibility: All users
4. Filter + Columns gemäß Spec
5. Save
6. Retrieve via `sf project retrieve start --metadata "ListView:..."`

### Akzeptanz für Track C

- Drei ListViews im Repo committet.
- In der UI sichtbar im ListView-Dropdown des jeweiligen Objekts.
- Jede ListView zeigt ausschließlich Demo-Records.
